# 프로젝트 개발 규칙 (공통 템플릿)

> 새 프로젝트에 복사해 사용하는 **공통 개발 규칙**입니다.  
> 스택·스키마·배포·도메인 정책 등 **프로젝트 전용 내용**은 이 문서를 복사한 뒤, 해당 레포 규칙 파일에 **추가·병합**합니다.

## 0) 기본 규칙

- 파일·모듈 작명은 **프로젝트 관례**를 따른다. (레포 기존 스타일 우선)
- 들여쓰기·포맷은 **프로젝트 설정**(EditorConfig, linter)에 맞춘다.
- 가독성을 위해 함수, 클래스 등 문단 간 **적절한 빈 줄**로 구분한다.
- 비자명한 로직에는 기능·의도를 설명하는 **짧은 주석**을 단다.
- 프로젝트는 **테스트 주도 개발(TDD)** 방식을 기본으로 한다.
  - Red-Green-Refactor

## 0-1) AI·코드 작성 원칙

> LLM 코딩(바이브 코딩) 시 흔한 오류를 줄이기 위한 행동 지침.

### 작성 전에 생각하기

- 멋대로 추측하지 않는다. 헷갈리는 점을 숨기지 않는다. 장단점을 명확히 드러낸다.
- **실행하기 전에**
  - 가정한 내용을 명확하게 밝히고, 확실하지 않으면 반드시 질문한다.
  - 요청을 여러 방법으로 해결할 수 있으면, 맘대로 선택하지 말고 각 방법의 장단점을 제시한다.
  - 더 간단한 방법이 있으면 언급한다. 필요하다면 반박한다.
  - 이해가 되지 않는 부분이 있으면 당장 멈추고, 무엇이 헷갈리는지 말하고 질문한다.

### 사전 제안·승인

- **새 파일 생성** 또는 **기존 파일 수정** 전에, 적용할 코드(또는 변경 요약)와 **어떤 기능/목적의 코드인지** 설명을 함께 제시한다.
- 사용자에게 **「이대로 작업을 진행해도 될까요?」** 등으로 확인을 받은 뒤 파일 생성·수정을 시작한다.
- **예외(즉시 실행 가능)**
  - 사용자가 **「바로 구현해줘」「적용해줘」「진행해줘」** 등 즉시 실행을 명시한 경우
  - **질문·설명만** 요청한 경우(`~가 뭐야?`, `왜 그래?`, `확인해줘`) — 조사·답변만 하고 파일은 건드리지 않는다.

### 단순함이 최우선

- 문제를 해결하는 데 **필요한 최소한의 코드만** 작성한다. 추측성 코드는 넣지 않는다.
- 변경된 모든 코드가 사용자 요청에 **직접 연결**되는지 확인하고, 연결되지 않으면 손대지 않는다.
- 일회용 코드에는 불필요한 추상화를 하지 않는다.
- 요청하지 않은 「유연성」「설정 가능성」을 추가하지 않는다.
- 불가능하거나 극히 드문 시나리오만을 위한 과도한 오류 처리를 하지 않는다.
- 200줄로 쓸 수 있는 것을 50줄로 줄일 수 있으면 다시 작성한다.
- 스스로에게 **「선임 개발자가 이것이 지나치게 복잡하다고 말할까?」**를 묻고, 그렇다면 단순화한다.

### 최소 변경

- 꼭 필요한 것만 수정한다. 자신이 저지른 일은 스스로 수습한다.
- 기존 내용을 **최대한 건드리지 않고**, 사용자 요청과 **직접 연결된 부분만** 작성·수정한다.
- 인접 코드·주석·서식을 멋대로 「개선」하거나, 멀쩡한 코드를 리팩터링하지 않는다.
- 기존 스타일·패턴을 따른다. (표현 방식이 달라도 프로젝트 기존 스타일을 우선한다.)
- 요청과 무관한 미사용 코드는 **바로 삭제하지 말고** 언급한 뒤 정리 여부를 묻는다.
  - 예: 「~에 대한 ABC 파일이 지금은 사용되지 않는 것 같습니다. 정리할까요?」
  - 예: 「get_user 함수에 안 쓰이는 import가 있는데 정리할까요?」
- **이번 변경으로 새로 생긴** 미사용 import/함수/변수는 제거한다.
- 요청받지 않은 한, **기존에 있던** 미사용 코드는 삭제하지 않는다.
- 변경·추가한 테스트도 사용자 요청과 직접 연결되어야 한다.

### 목표 기반 실행(TDD)

- 성공 기준을 정의하고, 기준이 충족될 때까지 반복한다.
- **Step 1)** 성공 케이스와 실패 케이스 각각의 테스트를 작성한다 → 검증: 현재 코드가 기대대로 실패하는지 확인한다. (Red)
- **Step 2)** 코드를 수정한다 → 검증: 실패 케이스로 작성한 테스트가 통과하는지 확인한다. (Green)
- **Step 3)** 회귀를 확인한다 → 검증: 수정 전에 통과하던 기존 테스트도 전부 통과하는지 확인한다. (Refactor)
- `0) 기본 규칙`의 TDD·`7) 테스트 규칙`과 함께 적용한다.

## 1) 아키텍처/보안

- **클라이언트(앱·웹)** 에 API 키·비밀값을 절대 하드코딩하지 않는다.
- 외부 API·AI 호출은 **서버(백엔드·BFF·Functions)** 에서만 수행한다.
- 서버 비밀값은 환경 변수·시크릿 매니저로 관리하고 Git에 커밋하지 않는다.
- 인증이 필요한 API는 **미인증 요청을 거부**한다.
- 환경별 설정은 **환경 변수**로 분리한다.

## 2) 데이터/시간 기준

- 날짜·시간이 비즈니스 로직에 쓰이면 **프로젝트에서 정한 기준 시간대**를 하나로 고정하고 일관 적용한다.
- 일일·주기 단위 키·조회·리셋 로직은 **동일한 날짜 함수**로 처리한다.
- 동시 갱신이 있는 데이터는 **트랜잭션·락·원자적 연산**을 우선하고, 상한·하한이 있으면 clamp 한다.

## 3) 클라이언트(프론트엔드) 코드 규칙

- 상태 변화가 있는 비동기 작업은 로딩/에러 상태를 명시적으로 관리한다.
- 화면 전환 후 요약·목록 데이터는 복귀 시 필요하면 재동기화한다(refresh).
- 디버그 전용 UI/기능은 **디버그 빌드·환경 플래그**로 감싸고, 운영 노출을 금지한다.
- 사용자 액션 기반 상태 변경은 **중복 탭·연타 방지**(1회 처리)를 유지한다.
- 온보딩·권한·필수 설정 미완료 시 진입 제어는 **프로필·설정 플래그**로 통제한다. (프로젝트별 상세는 레포 규칙에 정의)

## 3-1) i18n(로컬라이제이션) 규칙

- 사용자에게 노출되는 UI 문구는 하드코딩하지 않고 **프로젝트 i18n 체계**로 관리한다.
  - 대상: `Text`, `SnackBar`, `tooltip`, 폼 라벨, `AlertDialog`, 버튼 라벨, validator 메시지 등
- 새 문구를 추가할 때는 **모든 지원 로케일**에 키를 추가하고, 생성·빌드 절차를 따른다.
- 에러 메시지는 **i18n 문구 우선**, 필요 시 시스템/라이브러리 메시지를 fallback으로 사용한다.
- i18n 변경 후 프로젝트에 정의된 **품질 게이트**(생성, lint, test)를 통과한다.

## 4) 서버/백엔드 규칙

- 배포 리전·런타임은 **프로젝트 표준**을 따른다. (레포 규칙에 명시)
- DB·API 스키마·필드명은 **기존 데이터와 호환**되게 유지한다.
- 외부 API·AI 응답은 **스키마 검증** 후 사용하고, 실패 시 안전한 fallback 응답을 제공한다.

## 5) 품질/운영 규칙

- 변경 후 프로젝트에 정의된 **최소 품질 게이트**(lint, build, test 등)를 통과해야 한다.
- 한 번에 큰 리팩터링보다 기능 단위의 작은 변경/커밋을 우선한다.
- 진행 기록은 작업 종료 시점에 정리하되, **코드와 실제 동작**을 기준으로 작성한다.

## 6) 응답 규칙

- **`0-1) AI·코드 작성 원칙`을 우선한다.** 구현·수정 요청이어도, 즉시 실행 예외에 해당하지 않으면 **사전 제안·승인 후** 파일을 건드린다.
- 사용자의 요청 작업은 절대 누락하지 않는다. **승인을 받았거나** 즉시 실행 예외에 해당하면, 답변보다 실행(파일 수정/검증)을 먼저 완료한다.
- "요약 답변"만으로 작업을 종료하지 않는다. 사용자가 파일 반영을 요구한 경우 반드시 파일 반영 후 결과를 보고한다.
- 기능 수정 요청에 대해 설명·안내할 때는 **코드/저장소 관점만**으로 끝내지 않고, **실행·검증까지 포함한 완료 범위**(lint/test/build, 배포, 기존 데이터와의 차이, 수동 재현 절차)를 함께 짚는다. 에이전트가 대신 실행할 수 없는 단계는 **누가 어떤 명령으로 무엇을 확인하면 되는지** 순서대로 적는다.
- 기능 수정 이후에 사용자가 명령어를 직접 입력해야 하는 경우에는 입력할 명령어를 제시하여 안내한다.
- 기능 수정 이후에 테스트 방법을 순차적으로 정리하여 상세하게 안내한다.
- 코드 수정이나 작업 진행 시 API 발급이나 직접적인 명령어 입력 등 개발자의 직접적인 행동이 필요할 경우에는 그 과정을 상세하게 설명한다.
- 매 작업 이후 커밋이 필요하다고 판단되면 해당 작업에 맞는 커밋메시지를 추천하여 제공한다. (실제 커밋은 사용자 요청 시에만)
- 테스트마다 예상되는 성공, 실패 예시를 같이 제공하여 이후에 생기는 문제에 대해서도 예측할 수 있도록 한다.
- 개발자가 작업·세션 종료 요약을 요청하면, 프로젝트 **진행 기록 문서**에 정해진 템플릿 형식으로 이어서 작성한다.
- **진행 문서 누락 방지(에이전트·협업 공통)**
  - 작업·세션 종료 시점에 **요약·정리를 요청**하면, 채팅에만 적지 말고 **진행 기록 문서**에 단계별로 **반드시 append**한다.
  - 응답에는 **어느 파일·어느 단계**를 추가했는지 명시한다.
  - **트리거 예시:** `작업내용 정리`, `진행 기록 정리`, `오늘 한 일 정리`, `세션 마무리` 등. `정리해줘`만 단독으로 쓰면 의도가 모호할 수 있으므로, 가능하면 위 표현처럼 **진행 문서** 의도를 같이 적는 것이 좋다.

## 7) 테스트 규칙

- 기능 테스트 이후에 `console.log` 같은 **임시 로그**를 남기는 코드는 삭제한다.
- 테스트 전제(최소 품질 게이트)
  - 변경 영역에 맞는 lint·build·test를 프로젝트 표준에 따라 수행한다.

- 보안/인증 관련 필수 검증
  - 클라이언트·커밋에 API 키/비밀값이 포함되지 않았는지 점검한다.
  - 서버 API가 미인증 요청을 거부하는지 확인한다.
  - `.env`, credentials 등 비밀값 파일이 Git에 포함되지 않았는지 확인한다.

- 화면/상태(UX) 필수 검증
  - 비동기 작업은 로딩/성공/실패 상태가 UI에 명확히 반영되는지 확인한다.
  - 화면 복귀 시 필요한 데이터가 refresh되는지 확인한다.
  - 디버그 전용 UI/기능이 운영에 노출되지 않는지 확인한다.

- 외부 API·AI 응답 안정성 검증(해당 시)
  - 응답은 스키마 검증 후 사용되는지 확인하고, 실패 시 안전한 fallback이 동작하는지 확인한다.

## 8) Git 접근 규칙

- Push는 반드시 `7) 테스트 규칙`을 참고하여 검증과 테스트를 진행하고, 문제없이 통과한 이후에 진행한다.
- 사용자가 커밋을 요청하면, 현재 작업 상황을 토대로 논리 단위별로 나누어 커밋하고, 각 단위에 맞는 커밋 메시지를 작성한다.
- 사용자가 커밋(또는 커밋 메시지 작성)을 요청할 때만 커밋한다. 커밋 메시지 언어·형식은 **프로젝트 관례**를 따른다.

### 변경 되돌리기(undo) 요청 해석

- 사용자가 **「되돌려줘」「undo」「수정 전으로」「원래대로」** 등으로 요청할 때, **별도로 커밋·revert·reset을 명시하지 않았다면** Git **히스토리를 되돌리는 작업이 아니다.**
- 이 경우 의미하는 것은 **작업 트리(워킹 디렉터리)의 파일 내용만** 수정 전 상태로 복원하는 것이다.
  - 이미 커밋된 범위를 기준으로 되돌릴 때: 해당 파일에 대해 `git restore <path>` 등으로 **마지막 커밋(또는 지정한 기준) 내용으로 복원**
  - 이번 작업에서 **새로 만든 미추적(untracked) 파일**은 삭제
  - **요청 범위에 해당하는 파일만** 되돌리고, 무관한 변경은 건드리지 않는다.
- **하지 않는 것(명시 요청이 없으면 금지)**
  - `git revert`, `git reset --hard`, force push, 커밋 amend, 되돌림용 **새 커밋 생성**
- 사용자가 **「커밋 되돌려」「커밋 revert」「git 히스토리 되돌려」** 등 **Git 기록 조작**을 명시한 경우에만, Git 규칙·사용자 Git 규칙에 따라 히스토리 단위 작업을 수행한다.
- 되돌림 완료 후에는 **어떤 파일을 어떤 방식으로 복원했는지**(복원 / 삭제 / Git 히스토리 작업 여부)를 짧게 보고한다.

### 브랜치 운용 원칙

- `main`은 항상 **빌드/배포 가능한 상태**를 유지한다.
- 브랜치는 “유형을 미리 만들어두는 것”이 아니라, **작업이 생길 때마다 1개 생성 → 완료 후 머지 → (원격 포함) 삭제**를 기본으로 한다.
- 브랜치명은 소문자 + 하이픈을 사용하고, `type/topic`의 형태를 가진다. 예: `feature/auth-login`

### 어떤 상황에 어떤 브랜치에 커밋하는가

- `main/` : 작은 변경(단일 파일/짧은 수정, 리스크 낮음)의 간단한 수정
  - 예: README 오타, 문구/리소스 소폭 수정, 명확한 단일 버그 1건
- `feature/<topic>` : 기능 추가/화면 추가/흐름 변경(여러 파일, 1시간 이상 작업 가능성) 등 '새 기능'이 추가될 때
- `fix/<topic>` : 버그 수정/크래시/회귀(regression) 대응
  - 배포/로그인 불가 등 즉시 복구가 필요하면 `hotfix/<topic>` 사용 가능
- `refactor/<topic>` : 리팩터링/구조 개선
  - 동작 변화는 최소 + 구조/가독성/상태관리 개선
- `backend/<topic>` : API·DB·보안·인증·스키마 영향
- `ui/<topic>` : UI/디자인 수정
- `docs/<topic>` : 문서 작업
- `chore/<topic>` : 의존성/빌드/CI/설정 등 잡일성 작업
- `hotfix/<topic>` : 긴급 패치
  - `main`이 깨졌거나, 배포/로그인 불가 같은 즉시 복구가 필요할 때

### 브랜치 타입 예시

- `feature/home-summary-refresh`
- `fix/date-off-by-one`
- `refactor/user-repository`
- `backend/callable-auth-guard`
- `docs/workflow-reference`
- `chore/update-dependencies`

## 9) 작업 시작/종료 최소 체크(워크플로우 요약)

- 작업을 시작할 때는 “작업 유형”을 먼저 분류하고(클라이언트 / DB / 서버 / 문서), 아래 영향 범위를 체크한다.
  - **날짜·시간대** 영향 여부
  - **Auth/보안/secret** 영향 여부
  - **스키마 호환성** 영향 여부
- 구현 중에는 변경 단위를 작게 유지하고, 클라이언트에 외부 API 호출·비밀값이 들어가지 않게 유지한다.
- 종료(머지/푸시) 전에는 `7) 테스트 규칙`의 품질 게이트를 통과하고, 인증·스키마·핵심 도메인 로직 변경이면 핵심 플로우를 1회 이상 수동 검증한다.
